Sesuai ketentuan tugas (poin 10), berikut identifikasi risiko keamanan pada aplikasi ini beserta dampak dan mitigasinya.

AsetRisikoDampakLevelMitigasi
Dataset Dataset publik berisi ribuan URL yang sudah tidak aktif; tidak berisi PII namun tetap berisiko disalahgunakan sebagai referensi. Rendah — risiko re-identifikasi minim. Rendah Tidak menyimpan data tambahan pengguna aplikasi.
Source code File .pkl dari sumber tidak tepercaya berpotensi mengeksekusi kode berbahaya saat di-unpickle. Tinggi jika file model diganti pihak tidak berwenang. Tinggi Hanya memuat model dari repository terverifikasi; pertimbangkan checksum validation.
Fitur Cek URL (live fetch) Aplikasi melakukan HTTP request ke URL input pengguna — berpotensi SSRF jika mengarah ke alamat internal. Sedang/Tinggi — dapat membocorkan info jaringan internal. Tinggi Blok domain/IP privat (localhost, 127.0.0.1, 10.x, 192.168.x); batasi timeout & ukuran respons.
Dashboard/Aplikasi Jika dideploy publik tanpa autentikasi, siapa pun dapat memakai fitur Cek URL secara masif. Sedang — beban server berlebih, potensi disalahgunakan. Sedang Terapkan rate limiting per IP/sesi dan/atau autentikasi sederhana.
Model ML Model dapat bias terhadap pola dataset training dan tidak selalu general ke pola phishing baru (adversarial evasion). Sedang/Tinggi — false negative pada phishing canggih. Sedang Evaluasi berkala dengan data baru, kombinasikan dengan lapisan keamanan lain.
Input aplikasi Input URL tidak valid/format aneh dapat menyebabkan error atau permintaan HTTP menggantung. Sedang — aplikasi bisa crash/hang. Sedang Validasi format URL, batasi timeout request (sudah diterapkan: 10 detik).

Aspek Etika, Privasi, dan Bias Data

Privasi: aplikasi tidak menyimpan URL yang dicek pengguna secara permanen (hanya diproses sementara di memori sesi).

Bias: dataset training didominasi domain .com (~48%), sehingga model berpotensi kurang optimal pada TLD yang jarang muncul di data training.

Tanggung jawab penggunaan: hasil prediksi bersifat rekomendasi/skor risiko, bukan keputusan final.

Rencana Deployment, Optimasi, dan Continuous Improvement

Deployment: dijalankan lokal via python app.py, atau dideploy ke server kampus (mis. via Gunicorn + Nginx).

Optimasi: fitur visualisasi memakai sampel data (bukan seluruh 234 ribu baris); model dimuat sekali saat aplikasi start (bukan setiap request) agar tidak perlu training ulang.

Skalabilitas: jika volume data bertambah besar, pertimbangkan database terpisah (PostgreSQL) dan model-serving terpisah.

MLOps sederhana: version control kode via GitHub, requirements.txt untuk reproduksi environment, monitoring performa model secara berkala, dan rencana retraining periodik.
